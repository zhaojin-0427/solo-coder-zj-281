## 正式题目 Prompt

请基于当前项目完成一次“成分耐受建立与分阶段试用计划”模块的代码重构。该模块目前散落在 `backend/src/routes/tolerance.js`、`backend/src/utils/toleranceEngine.js`、`frontend/src/stores/tolerance.ts`、`frontend/src/views/ToleranceList.vue`、`frontend/src/views/ToleranceDetail.vue`、`frontend/src/components/Tolerance/*` 以及数据库迁移脚本中，存在路由层业务过重、字段映射不统一、计划生成和阶段评估耦合、前后端类型边界不清晰的问题。

你的目标不是新增功能，而是在不改变用户可见行为的前提下，重构耐受计划模块，让代码更易维护、更容易测试，并降低后续接入提醒中心、订阅配送内容和历史使用记录时的耦合成本。

### 必须完成的重构目标

1. 后端耐受计划模块需要拆分清楚职责：
   - 路由层只负责参数接收、错误响应和调用服务。
   - 计划 CRUD、阶段推进、每日反馈、中断/恢复、结果评估等业务流程需要集中到 service 层或等价结构。
   - SQLite 查询和写入需要收敛到 repository 层或等价结构。
   - 数据库 snake_case 与 API camelCase 的转换需要统一到 mapper/serializer 层或等价结构，不能散落在多个文件中。

2. 计划生成逻辑需要保持业务一致：
   - 支持 `7days`、`14days`、`custom` 周期。
   - 保留初始频率、滴数梯度、观察指标、阶段目标、暂停/延长判断条件。
   - 仍然要结合历史使用记录、皮肤敏感度、配方或成分风险提示生成建议。
   - 不能改变已有接口路径和前端调用方式，已有页面仍应可用。

3. 前端耐受计划相关类型和状态管理需要整理：
   - `frontend/src/types/index.ts` 中耐受计划相关类型要与后端响应结构保持一致。
   - `frontend/src/stores/tolerance.ts` 只能处理接口调用和本地状态更新，不要包含业务评估规则。
   - `ToleranceList.vue`、`ToleranceDetail.vue`、创建/反馈弹窗组件中的字段使用要统一为 camelCase。

4. 数据库迁移必须安全：
   - 不允许删除现有用户数据。
   - 不允许要求用户手动清空数据库。
   - 如果需要修正旧表结构或外键，必须通过幂等迁移脚本处理。
   - 多次运行迁移脚本不能导致重复字段、重复索引或数据丢失。

5. 保持现有项目约束：
   - 禁止引入新的第三方 npm 包。
   - 禁止重写整个项目或替换技术栈。
   - 禁止把耐受计划逻辑硬编码到前端页面。
   - 禁止为了通过测试而删除已有功能入口。

### 验收标准

1. 项目能正常启动：
   - 后端 `npm start` 正常运行在 `9102`。
   - 前端 `npm run dev` 正常运行在 `9101`。

2. 后端接口至少需要验证以下流程：
   - `GET /api/tolerance/preview` 能返回计划预览。
   - `POST /api/tolerance` 能创建计划并生成阶段。
   - `GET /api/tolerance/:id` 能返回计划、阶段、反馈和中断记录。
   - `POST /api/tolerance/:id/feedback` 能提交每日反馈，并返回暂停建议。
   - `POST /api/tolerance/:id/phase/:phaseId/evaluate` 能评估阶段。
   - `POST /api/tolerance/:id/phase/:phaseId/proceed` 能推进或延长阶段。
   - `POST /api/tolerance/:id/interrupt` 与 `POST /api/tolerance/:id/resume` 能完成中断和恢复。
   - `POST /api/tolerance/:id/evaluate` 能生成最终适配结论。
   - `GET /api/tolerance/stats/summary` 能返回统计汇总。

3. 前端页面需要保持可用：
   - 配方详情页和成分详情页的“开始耐受计划”入口仍能打开创建弹窗。
   - 耐受计划列表页能展示计划、筛选状态、显示统计卡片。
   - 耐受计划详情页能展示阶段进度、每日反馈、风险中断记录和最终适配建议。
   - 提醒中心中的耐受计划入口仍能跳转到 `/tolerance`。

4. 代码质量要求：
   - 路由文件明显变薄，复杂 SQL 和业务判断不应继续堆在路由处理函数中。
   - 字段转换逻辑只有一个明确入口，避免同一字段同时用 `source_type`、`sourceType` 在页面和 store 中混用。
   - 迁移脚本具备幂等性。
   - 新增或调整的核心函数应有清晰命名，避免大函数继续膨胀。

## 出题意图

这道题考察模型是否能读懂项目现有业务，而不是机械拆文件。耐受计划模块横跨前后端、数据库、历史记录、配方风险、提醒中心入口和页面交互，重构时很容易出现以下失误：

- 只移动代码但没有统一字段转换，导致前端部分页面读不到字段。
- 为了快速修复创建失败而清空数据库或重建表，破坏用户数据。
- 把业务规则搬到前端，导致接口和页面状态不一致。
- 改接口路径或响应结构，导致现有 Pinia store 和页面入口失效。
- 忽略中断/恢复、阶段推进、最终评估这类非首屏流程。
